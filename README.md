# Microseridor Scanner

Метод и средство обнаружения логических уязвимостей уровня доступа (IDOR) в распределенных информационных системах на основе анализа сетевого трафика

## Проект сканера IDOR в микросервисной архитектуре

### Модель развёртывания

Сканер предназначен для работы внутри кластера Kubernetes (например, как отдельный под со специальным сервисным аккаунтом).
Это обеспечивает:

- доступ ко всем внутренним REST API микросервисов (через их DNS-имена или ClusterIP);
- возможность перехватывать трафик между сервисами (если используется прокси или sidecar);

Альтернативный сценарий (для тестового стенда) — развёртывание в Minikube/Kind с полным контролем сети.
Внешнее сканирование через Ingress/Gateway также возможно, но тогда объектом анализа становится только публичное API — это частный случай, который не требует внутреннего доступа.

### Принцип поиска IDOR 

Строим карту запросов по kubernetes API по прокси серверу.

Мы находим те ID, которые жестко привязаны к пользователю по последовательности запросов.
Например, пусть formID - ID страницы анкеты в системе, которую заполняет пользователь. 
Какие-то POST-запросы будут возвращать это ID. 

Создаем три аккаунта: 2 аккаунта жертвы (Пользователь А, B) с разными параметрами и 1 злоумышленника:
1. Пользователи А и Б создают сущности с разными параметрами.
2. Пользователи А и Б вызывают проверяемый эндпоинт и получают тела ответов — legal_response_A и legal_response_B.
3. Злоумышленник последовательно подставляет идентификаторы А и Б и получает illegal_response_A, illegal_response_B.
4. Если illegal_response_A совпадает с legal_response_A (или содержит данные жертвы), illegal_response_B совпадает с legal_response_B, и при этом ответы не содержат маркеров отказа (403, "access denied"), то эндпоинт уязвим для IDOR.

### Основные компоненты

- Источник данных (вместо пассивного прокси снаружи):
  В реальной среде внутри кластера возможны два варианта:

Вариант А (упрощённый для тестового стенда):
Пассивный прокси, развёрнутый как sidecar-контейнер или отдельный под, который перехватывает трафик внутри кластера.
Прокси не является точкой входа снаружи, а просто слушает внутренние интерфейсы.

Вариант Б (рекомендуемый для production-подобной среды):
Использование сервис-меша (Istio + OpenTelemetry).
- Envoy-прокси автоматически собирают все запросы между сервисами.  
- Оркестратор читает логи Envoy или спаны Jaeger через API.

Это исключает необходимость в отдельном прокси и даёт полную наблюдаемость.

В описании диплома можно принять вариант А (прокси внутри кластера) как базовый, а вариант Б описать как перспективу.

- Построитель карты запросов (Модуль оркестратора?):

Модуль по kubernetes API и по OpenAPI строит связи между микросервисами, которую использует оркестратор.
Также анализирует информацию о запросах с пассивного прокси, верифицируя карту запросов, построенного с kubernetes API и сохраняя пример входящих параметров, для прохождения правил валидации.
Если каких-то эндпоинтов не хватает, то выполняем активное сканирование эндпоинтов.
K8s API даёт только топологию сервисов, а реальные эндпоинты и параметры извлекаются из пассивного прокси (и OpenAPI).

- Оркестратор

Центральный управляющий элемент.
Анализирует логи, классифицирует параметры (приватные/публичные) на основе поведенческих данных и эвристик.
При необходимости запускает точечное активное сканирование для уточнения.
Управляет созданием тестовых аккаунтов (через IdP).
Формирует план атаки, генерирует корректные запросы (на основе образцов из прокси), запускает активное сканирование, фиксирует результаты.
По окончании сессии сканирования (или при прерывании) Оркестратор проходит по списку created_resources и отправляет DELETE-запросы на соответствующие эндпоинты (если они поддерживаются).

- Активный сканер

Выполняет HTTP-запросы к API (напрямую, без прокси) согласно плану, анализирует ответы (сравнение с эталонами, возможно с применением ML) и возвращает результаты Оркестратору.

### Логика построения карты запросов
С прокси мы получаем информацию о существующих маршрутах.
Получив список эндпоинтов от Kubernetes API и Open API получаем список всех эндпоинтов в микросервисной архитектуре.
Связь между параметрами в теле ответа предыдущего запроса и потенциального последующего осуществляется с помощью семантического анализа с помощью модели ML.
Нужно было чтобы все обязательные параметры в запросе были закрыты предыдущими.

### Опциональное применение ML

Для повышения точности могут быть добавлены два ML-модуля:

- Классификатор параметров-идентификаторов
  Обучается на признаках (имя, тип значений, энтропия, позиция) на размеченной вручную выборке. Помогает автоматически определять, какие параметры являются идентификаторами, даже если они названы нестандартно.

- Семантический анализатор ответов (опционально, при доказанной эффективности)
  Использует предобученные модели (например, sentence-transformers/all-MiniLM-L6-v2) для сравнения ответов, устойчив к незначительным изменениям структуры. Может заменить или дополнить сравнение структуры.
  Как это будет работать?

1) Нормализация json-тела ответа
2) Векторизация тела ответа (набор чисел, описывающий смысл текста)
3) Расчет косинусное сходство (Cosine Similarity) между вектором ответа жертвы (legal_response) и вектором ответа злоумышленника (illegal_response).

Логика детектирования уязвимости:
Если illegal_response_A (ответ атакующего с ID жертвы А) имеет высокое семантическое сходство с legal_response_B (ответ жертвы А), и illegal_response_A имеет сходство с illegal_response_B, значит, атакующий получил те же данные. Это IDOR.
Если сходство низкое (например, атакующий получил ошибку "Access Denied" или пустой объект), значит, защита работает.
